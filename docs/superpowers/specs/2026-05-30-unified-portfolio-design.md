# Design Doc: Unified Portfolio (Trade Journal + Paper Trading)
**Date:** 2026-05-30
**Status:** Approved
**Topic:** Merging manual journaling with automated paper trading results into a single 'Portfolio' view.

---

## 1. Problem Statement
Currently, the application has two separate features for tracking performance:
1. **Trade Journal:** For manual entry of trades.
2. **Paper Trading:** For automated tracking of signals generated by the screening engine.

Users have to check two different pages to see their total performance, and automated trades don't benefit from the journaling features (notes, manual close overrides, etc.).

## 2. Proposed Solution: "Hybrid Sync"
We will consolidate these into a single **Portfolio** system where:
- **TradeJournal** becomes the "Master Ledger" for all finalized and open positions.
- The **Paper Trading Engine** acts as an automated "user" that logs its trades into the Journal.
- The UI provides a unified dashboard to compare "System" performance vs "Manual" performance.

---

## 3. Architecture & Data Model

### 3.1 Database Updates (`TradeJournal` table)
Add the following columns to the `trade_journal` table:
- `source`: `String` (Enum: `'manual'`, `'paper'`) - Defaults to `'manual'`.
- `external_id`: `Integer` (Nullable) - Stores the `PaperPosition.id` for synced trades to prevent duplicates and allow updates.

### 3.2 Sync Logic (Backend)
The `PaperTradingEngine` will be updated to sync with the `TradeJournal`:
1. **On Position Entry:** When a paper trade moves from `pending` to `open`, create a `TradeJournal` entry with `source='paper'`.
2. **On Position Update:** Daily price updates for paper positions will sync to the Journal entries to keep "Current Value" and "Unrealized P&L" accurate.
3. **On Position Exit:** When a paper trade closes, update the Journal entry with `exit_price`, `exit_date`, `exit_reason`, and `status='closed'`.

---

## 4. User Interface (The "Portfolio" Page)

### 4.1 Views
- **Summary Cards:** Total Equity, Manual ROI, System ROI, Combined Win Rate.
- **Active Positions Table:** Mixed list of Manual and System trades, sortable by source.
- **Trade History Table:** Mixed list of closed trades.

### 4.2 Filtering
- Filter toggle: `[ All ] [ Manual ] [ System ]`
- This allows the user to see how they are doing compared to the automated system.

---

## 5. Implementation Steps

1. **Database:** Create Alembic migration for `trade_journal` columns.
2. **Backend Logic:**
    - Update `TradeJournal` model in `models.py`.
    - Create a `sync_service.py` or update `journal.py` to handle automated entries.
    - Update `PaperTradingEngine` to call the sync logic.
3. **API:** Update journal endpoints to support filtering by `source`.
4. **Frontend:**
    - Refactor `Journal.jsx` into `Portfolio.jsx`.
    - Add "Source" badges and filter toggles.
    - Update dashboard navigation.
    - Redirect old `/journal` and `/paper` routes to the new `/portfolio`.

---

## 6. Testing Strategy
- **Unit Test:** Verify `PaperTradingEngine` correctly triggers journal entries on trade events.
- **Integration Test:** Ensure updating a paper position correctly reflects in the Journal's unrealized P&L.
- **Manual Test:** Create a manual trade, run the pipeline to trigger a paper trade, and verify both appear in the unified Portfolio page.
