# Design Spec: Robust Stop-Loss Anchoring & Entry Filters

**Date:** 2026-06-08
**Topic:** Fixing "Zombie Stops" and high-volatility entry failures in the Backtest Engine.

## 1. Problem Statement
The current backtest engine suffers from two critical flaws:
1.  **Zombie Stops:** Stop-loss prices are anchored to the `Signal Close` price. When entering on a pullback days or weeks later, the actual risk relative to the `Entry Price` can balloon to 30%+, far exceeding the configured 2% risk-per-trade.
2.  **Falling Knives:** The engine enters at the Open of the entry day even if the stock immediately crashes through the stop-loss price in the same session, leading to "instant" stop-outs that could be avoided.
3.  **Explosive Signals:** Signals generated by bars with extreme intraday range (relative to the stop distance) are high-risk and often lead to immediate reversals or "noise-outs."

## 2. Proposed Changes

### 2.1 Configuration Updates (`UnifiedTradingConfig`)
- Add `initial_stop_atr_multiplier` (float, default: `2.0`): Multiplier for ATR used *only* for the initial stop placement at entry.
- Maintain `atr_multiplier` (default: `2.5`) for trailing stops and risk-reward calculations.

### 2.2 Stop-Loss Anchoring Refactor (`engine.py`)
The `stop_price` calculation will move from the signal-generation phase to the **entry-execution phase**.

**Logic:**
```python
# Structural floor from consolidation (e.g., 10-bar low)
struct_stop = consol_low * 0.98 if consol_low else 0

# Volatility floor relative to ACTUAL execution price
vol_stop = entry_price - (config.initial_stop_atr_multiplier * eff_atr)

# Hard cap floor relative to ACTUAL execution price
hard_cap_stop = entry_price * (1 - config.stop_loss_pct / 100)

# Final Stop Price: The tightest (highest price) of the three
# This ensures we respect technical structure, volatility, AND the hard risk limit.
stop_price = max(struct_stop, vol_stop, hard_cap_stop)
```

### 2.3 Entry Sanity Filters
Two new "gates" will be added to the trade simulation loop:

1.  **Signal Bar Volatility Filter:**
    - Skip signal if `(Signal High - Signal Low) > (Intended Stop Distance * 1.5)`.
    - *Rationale:* Prevents chasing "blown out" candles where the rubber band is overextended.

2.  **Entry Day "Falling Knife" Filter:**
    - Skip entry if `Entry Day Low <= Stop Price`.
    - *Rationale:* If the stop is hit on the very day we were supposed to enter, we treat it as a "no-trade" rather than an "instant loss."

## 3. Implementation Details

### 3.1 `backend/app/core/trading_config.py`
- Add `initial_stop_atr_multiplier` to the `UnifiedTradingConfig` dataclass.

### 3.2 `backend/app/backtest/engine.py`
- Refactor the `simulate_trades` function to defer `stop_price` calculation until the `entry_idx` is reached.
- Implement the `signal_bar_range` check immediately after a signal is identified but before waiting for a pullback/entry.
- Implement the `entry_day_low` check inside the entry execution block.

## 4. Verification Plan

### 4.1 Automated Tests
- **Test Case: Anchor Integrity:** Verify that a 7% `stop_loss_pct` results in exactly a 7% loss (minus slippage/commission) regardless of the gap between signal and entry.
- **Test Case: Falling Knife:** Provide a synthetic OHLCV where the entry day low breaches the stop; verify no trade is recorded.
- **Test Case: Signal Volatility:** Provide a massive signal bar; verify signal is ignored.

### 4.2 Backtest Comparison
- Run a benchmark backtest on the "NSE 500" universe.
- Compare `max_drawdown_pct` and `avg_loss_pct` before and after.
- **Success Criteria:** `avg_loss_pct` should be capped strictly by `config.stop_loss_pct` (plus minimal gap slippage), and the -31% "Zombie Stop" outliers should be eliminated.
